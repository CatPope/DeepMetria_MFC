## 1. 프로젝트 개요 ⏱ 1:00

> 🎯 비기술자엔 공감, 기술자엔 문제·접근의 일관성

**1-1. 문제**
- 마케터·기획자·경영진은 매일 데이터를 보지만 직접 분석은 막힌다 (엑셀·BI 학습 장벽, 개발자 의존)
- 🎤 "데이터는 누구나 갖고 있는데, 분석할 사람은 따로 있습니다."

**1-2. 솔루션 + 스택**
- **"파일을 올리고 말로 질문하면, AI가 차트로 답한다."** (📁→🤖→💬→📊)
- 비기술 한 줄: 코드 없이, 내 PC에서 쓰는 데이터 분석 앱
- 기술 한 줄: MFC(C++) · SQLite · WinHTTP · GDI+ 기반 Windows 네이티브 앱, 멀티 LLM 지원

🎨 4단계 플로우 + 스택 한 줄

## 2. 프로젝트 수행 — 핵심 설계와 판단 ⏱ 2:00 (비중 최대)

> 🎯 채점의 분기점. 같은 내용을 두 청중 언어로 동시에 전달.

**2-1. 아키텍처 — AI 오케스트레이터**
```
사용자(MFC) → AI 오케스트레이터(CoT) → LLM Router ─→ Claude/GPT/Gemini
            → 분석 도구(검증된 C++ 함수) → GDI+ 차트 → 대시보드
```
- 🎤 "AI는 지휘자일 뿐, 계산은 검증된 코드가 합니다."

**2-2. 판단 ① 믿을 수 있는 AI — "숫자를 지어내지 않는다" (핵심)**
- 비기술: AI는 똑똑한 비서, 계산은 정확한 계산기 → 결과를 믿을 수 있다
- 기술: LLM은 도구 선택·인자 결정만, 계산은 검증된 함수가 수행. 화이트리스트 내 선택 + 인자 검증 → 환각 차단, 결정성 확보

**2-3. 판단 ② 멀티 LLM Router**
- 비기술: 여러 AI(Claude·GPT·Gemini)를 갈아끼울 수 있어 한 곳이 막혀도 계속 작동
- 기술: 공통 인터페이스로 요청/응답 정규화, 프로바이더 어댑터가 포맷 차이 흡수(FR-08), 장애 시 폴백

**2-4. 판단 ③ 반응성 & 보안**
- 반응성(기술): LLM 응답 최대 60초 → 워커 스레드 + `WM_USER`로 UI 프리즈 방지
- 보안(공통): 데이터는 로컬 저장, API 키는 DPAPI 암호화 보관(평문 미노출)

🎨 아키텍처 도식 + "비서↔계산기" 강조
