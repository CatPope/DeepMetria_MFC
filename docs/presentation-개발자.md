## 1. 프로젝트 개요 ⏱ 1:00

> 🎯 "문제 정의 → 해결 접근 → 기술 선택"의 일관성을 먼저 각인

**1-1. 문제와 솔루션**
- 비개발자(마케터·기획자)는 데이터는 있으나 분석 수단이 없다 → 코드/BI 학습 장벽
- **DeepMetria**: 파일을 올리면 AI가 자동 요약하고, 자연어 질문을 검증된 분석 도구로 처리해 GDI+ 차트로 시각화하는 Windows 데스크톱 앱

**1-2. 기술 스택 & 제약**
| 영역 | 선택 | 이유 |
|------|------|------|
| GUI | MFC (C++) | 네이티브 성능, 단일 실행 파일 배포, GDI+ 직접 제어 |
| 저장 | SQLite | 로컬 단일 사용자, 임베디드 트랜잭션 |
| 통신 | WinHTTP | 외부 의존성 없는 LLM API 호출 |
| 렌더링 | GDI+ | 차트 직접 렌더링 |
- 제약: 단일 사용자 로컬 MVP, 멀티 LLM 추상화 필수, 평문 키 디스크 저장 금지

## 2. 프로젝트 수행 — 핵심 설계와 기술적 판단 ⏱ 2:00 (비중 최대)

> 🎯 기술 검토 점수의 핵심. "왜 이렇게 설계했는가"를 트레이드오프로 설명.

**2-1. 아키텍처 — AI 오케스트레이터 모델**
```
사용자(MFC View) → AI 오케스트레이터(CoT)
   → LLM Router ──→ Claude / GPT / Gemini
   → AnalysisTool 세트(검증된 C++ 함수, 직접 호출/COM)
   → GDI+ 렌더러 → 대시보드 View (WM_USER 갱신)
```
- 🎤 "LLM은 오케스트레이터일 뿐, 계산 주체가 아닙니다."

**2-2. 판단 ① 신뢰성 — LLM이 계산하지 않는다**
- 문제: LLM 직접 계산 시 숫자 환각(hallucination)
- 설계: LLM은 **도구 선택 + 인자 결정**만, 실제 계산은 검증된 함수가 수행
- 안전장치: 미리 정의된 도구 화이트리스트 내에서만 선택, 인자 검증 후 호출
- 효과: 결과 수치의 결정성·정확성 확보

**2-3. 판단 ② 멀티 LLM Router 추상화**
- 공통 인터페이스로 요청/응답 정규화 → 프로바이더별 어댑터가 포맷 차이 흡수
- 함수 호출 스키마·토큰·에러 형식 차이를 Router 레이어에서 캡슐화
- 효과: 모델 교체가 기능에 영향 없음(FR-08), 장애 시 폴백 가능

**2-4. 판단 ③ 반응성 — UI 스레드 보호**
- 문제: LLM 응답 최대 60초, 동기 호출 시 UI 프리즈
- 설계: 워커 스레드에서 WinHTTP 호출 → `WM_USER` 메시지로 메인 스레드 뷰 갱신
- 진행 표시·취소 처리 포함

**2-5. 판단 ④ 키 보안**
- API 키는 DPAPI로 암호화해 `HKCU\Software\DeepMetria\Settings`에 저장
- 평문 키를 소스·설정·로그에 남기지 않음 (사용자 계정 단위 복호화)

🎨 아키텍처 도식 + 2-2 "두뇌↔도구 분리" 강조
