# DeepMetria 핵심 설계

> 본 문서는 DeepMetria MVP의 **핵심 아키텍처와 설계 판단**을 정리한다.
> 근거 요구사항: `docs/SRS.md` (FR-01~08, NFR-01~03). 발표·검토 시 기술 설명의 기준 문서로 사용한다.

---

## 1. 설계 원칙

| 원칙 | 내용 |
|------|------|
| **AI는 판단, 계산은 코드** | LLM은 분석 계획·도구 선택만 담당하고, 실제 수치 계산은 검증된 C++ 함수가 수행한다 (숫자 환각 차단). |
| **프로바이더 독립** | 특정 LLM에 종속되지 않도록 Router 추상화 레이어로 멀티 LLM을 지원한다. |
| **UI 무중단** | 최대 60초의 LLM 응답 동안에도 UI가 멈추지 않는다 (비동기 처리). |
| **로컬 우선·키 보호** | 데이터는 로컬에 저장하고, API 키는 평문으로 디스크에 남기지 않는다. |
| **네이티브 단순 배포** | 외부 런타임 의존을 최소화한 단일 실행 파일 배포(MFC + WinHTTP + GDI+). |

---

## 2. 전체 아키텍처 — AI 오케스트레이터 모델

```
┌─────────────────────────────────────────────────────────────┐
│  사용자 (MFC UI)                                              │
│   - CFileDialog(파일 열기)  - CEdit(자연어 질문)             │
│   - 대시보드 CView          - CListCtrl(표)                  │
└───────────────┬─────────────────────────────────────────────┘
                │ 직접 함수 호출
                ▼
┌─────────────────────────────────────────────────────────────┐
│  AI 오케스트레이터 (Chain of Thinking)                       │
│   질문 → 분석 계획 수립 → 도구 선택 → 인자 결정 → 결과 종합  │
└───────────────┬─────────────────────────────────────────────┘
                │
        ┌───────┴────────┐
        ▼                ▼
┌───────────────┐  ┌─────────────────────────────────────────┐
│ LLM Router    │  │ AnalysisTool 세트 (검증된 C++ 함수)      │
│ (추상화)      │  │  - 통계 계산  - 집계/그룹  - 추세/이상치  │
│  ↓ WinHTTP    │  │  ※ 화이트리스트, 직접 호출 / COM         │
│ Claude/GPT/   │  └───────────────┬─────────────────────────┘
│ Gemini        │                  │ 분석 결과
└───────────────┘                  ▼
                          ┌─────────────────────────┐
                          │ GDI+ 차트 렌더러        │
                          │  → Visualization 생성   │
                          │  → 대시보드 (WM_USER)   │
                          └─────────────────────────┘
            ┌──────────────────────────────────────┐
            │ SQLite (DataSource / 대시보드 영속화) │
            │ 레지스트리(HKCU) + DPAPI (API 키)     │
            └──────────────────────────────────────┘
```

핵심: **LLM은 계산 경로 밖에 있다.** LLM의 출력은 "어떤 도구를 어떤 인자로 호출할지"이며, 수치는 항상 검증된 함수가 만든다.

## 3. 핵심 컴포넌트

| 컴포넌트 | 책임 | 기술 |
|----------|------|------|
| **MFC UI** (CMainFrame / CView / 컨트롤) | 파일 열기, 질문 입력, 대시보드 렌더, 서식 편집, 내보내기 | MFC |
| **AI 오케스트레이터** | CoT 추론으로 분석 계획 수립 → 도구 선택·인자 결정 → 결과 종합 | C++ |
| **LLM Router** | 멀티 LLM 추상화. 요청/응답 정규화, 프로바이더 어댑터, 폴백 | WinHTTP |
| **AnalysisTool 세트** | 실제 계산 수행(통계·집계·추세·이상치 등). 화이트리스트로 노출 | C++ (함수/COM) |
| **GDI+ 차트 렌더러** | 분석 결과 → 차트/표 시각화, 이미지 내보내기 | GDI+ |
| **영속화 계층** | DataSource·대시보드 상태 저장, 트랜잭션 | SQLite |
| **보안 계층** | API 키 DPAPI 암호화 저장/복호화 | DPAPI + 레지스트리(HKCU) |

## 4. 핵심 데이터 모델 (개념)

| 객체 | 설명 | 주요 필드(개념) |
|------|------|-----------------|
| **DataSource** | 불러온 데이터 원천 | id, 원본 경로/형식, 스키마, 행 수 |
| **DataSummary** | AI 자동 요약 | 컬럼 스키마(이름·타입), 도메인 추정, 기본 통계(행수·결측·수치통계) |
| **AnalysisFlow** | 자연어 질문 1건의 분석 처리 | 질문 텍스트, CoT 계획, 선택된 도구, 결과 |
| **AnalysisTool** | 분석 도구 정의 | 이름, 설명, 인자 스키마(LLM에 노출), 실행 함수 |
| **Visualization** | 시각화 요소 | 차트 유형, 데이터, position/size/style |
| **Dashboard** | 시각화 모음 뷰 | Visualization 목록, 레이아웃 |

## 5. 주요 흐름

### 5-1. 파일 불러오기 → 자동 요약 (FR-01, FR-02 / NFR-01 ≤30초)
1. CFileDialog로 파일(CSV/Excel/JSON) 선택
2. 형식 검증·파싱 → DataSource 생성 → SQLite 저장 (실패 시 트랜잭션 롤백, NFR-03)
3. LLM이 CoT로 스키마 파악 → 도메인 분류 → 기본 통계 **도구 호출**
4. DataSummary 생성 → 기본 대시보드 CView 구성

### 5-2. 자연어 질문 → 시각화 (FR-03~05 / NFR-01 ≤60초)
1. CEdit에 질문 입력 → AnalysisFlow 생성, 진행 표시 시작
2. **워커 스레드**: LLM Router로 질문 전달 → CoT 추론 → 도구 선택·인자 결정
3. 선택된 AnalysisTool을 **인자 검증 후** C++ 함수로 호출 → 결과 수신
4. AI가 차트 유형 결정 → GDI+ 렌더 → Visualization 생성
5. `WM_USER` 메시지로 **메인 스레드** 대시보드 갱신

### 5-3. 편집·내보내기 (FR-06, FR-07)
- 마우스 메시지로 position/size/style 변경 → SQLite 저장 → 즉시 반영
- GDI+로 이미지(PNG/BMP) 렌더 → CFileDialog로 저장

## 6. 핵심 설계 판단 (트레이드오프)

### 판단 ① 신뢰성 — LLM을 계산 경로에서 분리
- **문제**: LLM 직접 계산 시 숫자 환각(hallucination)으로 결과 신뢰 불가
- **결정**: LLM은 도구 선택·인자 결정만, 계산은 검증된 함수가 수행
- **안전장치**: 미리 정의된 도구 화이트리스트 내에서만 선택, 인자 스키마 검증 후 실행
- **효과**: 결과 수치의 결정성·정확성 / **비용**: 도구 세트를 사전 구현해야 함(확장은 도구 추가)

### 판단 ② 멀티 LLM Router 추상화 (FR-08)
- **문제**: 프로바이더마다 함수 호출 포맷·토큰·에러 형식이 다름
- **결정**: 공통 인터페이스로 요청/응답 정규화, 프로바이더별 어댑터가 차이 흡수
- **효과**: 모델 교체가 기능에 무영향, 장애·쿼터 초과 시 폴백 / **비용**: 어댑터 유지보수

### 판단 ③ 반응성 — UI 스레드 보호
- **문제**: LLM 응답 최대 60초, 동기 호출 시 UI 프리즈
- **결정**: 워커 스레드에서 WinHTTP 호출 → `WM_USER`로 메인 스레드 갱신
- **효과**: 분석 중에도 UI 반응 유지, 진행 표시·취소 가능 / **비용**: 스레드 간 동기화 관리

### 판단 ④ 키 보안
- **문제**: 평문 API 키가 소스·설정·로그에 노출되면 안 됨
- **결정**: DPAPI로 암호화해 `HKCU\Software\DeepMetria\Settings`에 저장(사용자 계정 단위 복호화)
- **효과**: 디스크 평문 키 제거 / **비용**: 사용자 프로파일 종속(다중 사용자 PC는 post-MVP 고려)

### 판단 ⑤ MFC + GDI+ 네이티브 선택
- **결정**: Electron/웹뷰 대신 MFC 네이티브 + GDI+ 직접 렌더링
- **효과**: 네이티브 성능, 단일 실행 파일 배포, 차트 렌더 직접 제어
- **비용**: 현대적 UI/개발 생산성에서 불리 (트레이드오프 인지)
